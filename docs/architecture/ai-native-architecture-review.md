# 新点 SaaS 造价系统 AI 原生架构复盘与补强建议

> 这份文档用于在现有业务架构和技术架构基础上，补充“对主流 AI 工具友好”的系统设计视角，重点关注 `MCP`、`skills`、系统记忆、知识库与知识图谱能力。

## 1. 复盘目标

当前系统已经完成了：

- 业务设计
- 技术架构选型
- 清单表格设计
- 流程与表单引擎设计
- 部署架构设计

但如果从“AI 原生友好”的角度看，现有方案仍然偏：

`AI 功能接入型系统`

而不是：

`AI 可长期增智型系统`

两者差别很大。

前者只是：

- 调用模型
- 生成推荐
- 写一张 `ai_recommendation` 表

后者则要求系统本身能够：

- 被 AI 工具稳定调用
- 把业务数据沉淀成可复用知识
- 把流程经验沉淀成技能
- 让 AI 记住项目上下文、组织知识和历史判断
- 随着数据积累提升 AI 推荐质量

## 2. 当前方案的优点

从 AI 原生视角看，现有方案已经有 5 个很好的基础。

### 2.1 业务主链结构化程度高

当前系统已经有清晰的：

- 项目
- 阶段
- 清单版本
- 定额
- 审核
- 锁定
- 审计

这些结构化主数据，是 AI 能力升级的前提。

### 2.2 审计日志基础很好

你已经要求：

- 关键状态切换写审计日志
- AI 推荐接受/忽略写审计日志
- 导入、重算、审核都有追溯

这对后续做：

- 经验抽取
- 决策学习
- 失败案例识别

非常重要。

### 2.3 流程和表单已经被结构化

这意味着后续可以把：

- 审核路径
- 节点意见
- 驳回原因
- 单据类型差异

沉淀成 AI 可利用的流程知识。

### 2.4 清单与定额主数据是规范化存储

这为后续做：

- 检索增强
- 相似清单推荐
- 相似定额推荐
- 历史项目对比

提供了坚实基础。

### 2.5 已经有项目复盘和指标库概念

业务层已经明确了：

- 项目复盘
- 指标提取
- 历史资产沉淀

这说明产品目标本身就支持“越用越聪明”。

## 3. 当前方案的主要缺口

如果按 AI 原生系统标准看，当前最大缺口有 6 个。

## 3.1 缺少正式的 MCP 能力层

目前系统是“面向人类前端页面 + REST API”设计的。  
但如果要和主流 AI 工具深度耦合，系统还需要一层：

- `MCP Server Capability Layer`

也就是：

- 让 AI 能按稳定协议读取项目、阶段、清单、审核、报表、指标
- 让 AI 能拿到结构化上下文，而不是只能模拟用户点页面

### 目前的问题

当前 API 契约适合前后端联调，但不等于适合 AI Agent 直接使用。

因为 AI 更需要：

- 面向任务的资源组织
- 面向语义的上下文打包
- 面向权限的裁剪结果

## 3.2 缺少技能层 `skills layer`

现在系统还没有明确区分：

- 基础模型能力
- 可复用业务技能

例如在这个系统里，其实未来会自然长出很多“技能”：

- 清单差异解释技能
- 定额推荐解释技能
- 变更单审核建议技能
- 结算偏差复盘技能
- 导入纠错技能

如果没有 skills 层，AI 只能每次重新临场推理，无法稳定积累业务方法。

## 3.3 缺少系统记忆层 `system memory`

当前数据里有审计日志和推荐记录，但这还不等于“记忆”。

AI 记忆至少要分层：

- 用户级记忆
- 项目级记忆
- 组织级记忆
- 流程级记忆

比如：

- 某个项目一贯使用的定额风格
- 某类项目常见驳回原因
- 某地区造价员常用判断规则
- 某组织偏好的取费口径

这些如果不抽象成记忆层，AI 每次还是像第一次见项目一样。

## 3.4 缺少知识沉淀层

目前“项目复盘”“指标库更新结果”“历史造价数据库资产”这些概念都有，但还没真正设计：

- 如何抽取知识
- 如何存储知识
- 如何让 AI 使用这些知识

换句话说，当前还没有：

- 知识库结构
- 知识文档抽取规则
- 经验规则条目化机制
- 复盘结论结构化存储机制

## 3.5 缺少知识图谱视角

当前模型已经天然适合做图谱关系，例如：

- 项目 -> 阶段 -> 版本
- 版本 -> 清单项 -> 定额
- 清单项 -> 过程单据
- 审核记录 -> 驳回原因
- 项目复盘 -> 偏差类型 -> 指标

但当前还没有明确把这些关系当作“可推理知识图”来设计。

即使 V1 不上真正图数据库，也应该先把图谱语义建模出来。

## 3.6 缺少 AI 能力闭环定义

现在 AI 更多是：

- 推荐
- 预警

但还没有完整闭环：

`数据 -> 推荐 -> 人工处理 -> 审计 -> 复盘 -> 知识沉淀 -> 下一轮更准`

这是 AI 友好系统最关键的地方。

## 4. 建议新增的 AI 原生能力分层

建议把系统补成 5 层 AI 原生结构。

## 4.1 Resource Layer

也就是当前已有的业务资源层：

- 项目
- 阶段
- 清单
- 定额
- 报表
- 审核
- 审计

这一层你已经基本做得很好。

## 4.2 MCP Capability Layer

建议新增一层 MCP 能力层，对 AI Agent 暴露结构化资源。

推荐至少暴露这些 MCP 能力：

- `get_project_context`
- `get_stage_context`
- `get_bill_version_context`
- `get_quota_context`
- `get_review_context`
- `get_variance_context`
- `search_historical_cases`
- `search_knowledge_entries`
- `list_pending_work_items`

### 原则

MCP 层不一定等于数据库，也不一定等于 REST API。  
它应该是“为 AI 打包过的上下文入口”。

## 4.3 Skills Layer

建议显式设计一层业务技能。

推荐先规划这些技能类型：

- `bill-audit-skill`
- `quota-recommend-skill`
- `variance-analysis-skill`
- `change-order-review-skill`
- `settlement-retrospective-skill`
- `import-cleanup-skill`

### 技能的本质

技能不是模型，而是：

- 明确输入
- 明确上下文来源
- 明确执行步骤
- 明确输出结构

这样 AI 能更稳定地完成业务任务。

## 4.4 Memory Layer

建议补一层系统记忆设计。

推荐分 4 类：

### 用户记忆

- 用户常用专业
- 常用筛选方式
- 常用报表模板

### 项目记忆

- 项目历史异常点
- 阶段审核关注点
- 项目常见差异说明

### 组织记忆

- 企业内部审核偏好
- 组织常用定额处理方式
- 常见驳回规则

### AI 运行记忆

- 某类推荐经常被接受/忽略
- 某类输入导致高错误率
- 某类流程节点高频驳回原因

## 4.5 Knowledge Layer

建议把知识沉淀分成两类：

### 文档型知识库

来源：

- 项目复盘
- 审核意见
- 结算说明
- 标准指引
- 企业规则

### 结构化知识库

来源：

- 指标库
- 偏差分类
- 驳回原因分类
- 专业经验规则
- 定额适用模式

### V1 最好做到

至少把这些内容结构化出来：

- 复盘结论
- 驳回原因标签
- AI 推荐接受率
- 偏差原因分类

## 5. 建议新增的数据层设计

在现有数据模型上，建议未来补这些实体。

## 5.1 `knowledge_entry`

用于承接结构化知识条目。

建议字段：

- `id`
- `knowledge_type`
- `title`
- `summary`
- `source_type`
- `source_id`
- `project_id`
- `stage_code`
- `discipline_code`
- `tags`
- `content_json`
- `status`
- `created_at`

## 5.2 `memory_entry`

用于承接用户/项目/组织/AI 运行记忆。

建议字段：

- `id`
- `memory_scope`
- `scope_id`
- `memory_type`
- `summary`
- `payload_json`
- `importance_score`
- `last_used_at`
- `created_at`

## 5.3 `skill_definition`

用于登记系统内的业务技能。

建议字段：

- `id`
- `skill_code`
- `skill_name`
- `input_schema`
- `output_schema`
- `prompt_template`
- `resource_dependencies_json`
- `status`

## 5.4 `knowledge_relation`

用于表达轻量知识图谱关系。

建议字段：

- `id`
- `from_type`
- `from_id`
- `relation_type`
- `to_type`
- `to_id`
- `weight`

即使 V1 不上 Neo4j，也可以先在关系库里表达图关系。

## 6. 对现有架构的具体补强建议

## 6.1 技术架构补强

当前技术架构建议新增一个逻辑层：

`API App + Worker + MCP Capability Service + Knowledge/Memory Pipeline`

不一定一开始拆成独立服务，但在代码组织上应该预留模块。

推荐新增模块：

- `mcp-capability`
- `knowledge-pipeline`
- `memory-service`
- `skill-runtime`

## 6.2 工作流引擎补强

流程引擎里建议补：

- 节点意见结构化标签
- 驳回原因标签
- 审核意见抽取 pipeline

这样流程数据才能变成知识资产，而不只是留在 comment 文本里。

## 6.3 清单表格补强

清单表格相关动作建议额外记录：

- 用户改了哪些字段
- 最终接受了哪类 AI 推荐
- 哪些校验警告被反复忽略

这些都是极强的训练与复盘信号。

## 6.4 审计日志补强

当前审计日志已经不错，但如果要服务 AI，建议补：

- `decision_reason`
- `source_context_snapshot`
- `ai_assist_trace_id`

这样 AI 才能回头理解：

- 为什么这次接受了推荐
- 为什么那次驳回了
- 当时参考了什么上下文

## 7. 推荐的 AI 增智闭环

推荐正式定义下面这条闭环：

1. 用户在业务系统中操作
2. 系统写业务数据与审计日志
3. AI 生成推荐或预警
4. 人工接受、忽略、驳回
5. 系统记录结果与原因
6. 复盘任务抽取知识和记忆
7. 知识条目和记忆条目进入可检索层
8. 下次技能执行时优先使用这些沉淀

这条闭环才是“数据越多，AI 越强”的真正路径。

## 8. 对主流 AI 工具亲和力的具体要求

如果你要让系统以后对主流 AI 工具更友好，建议明确这些要求。

## 8.1 对 MCP 友好

- 资源上下文能被打包输出
- 支持按项目/阶段/对象获取结构化上下文
- 支持搜索历史案例与知识条目
- 支持读取待办、审核、差异、复盘数据

## 8.2 对 Skills 友好

- 技能输入输出边界清晰
- 技能依赖资源清晰
- 技能执行结果可追溯
- 技能可以挂接组织记忆和项目记忆

## 8.3 对 Memory 友好

- 记忆不是硬编码在 prompt 里
- 记忆可更新、可衰减、可检索
- 记忆作用域清楚

## 8.4 对知识库/图谱友好

- 文档型知识和结构化知识并存
- 关系链可追溯
- 支持从复盘、审核、异常中自动抽取

## 9. 建议优先级

如果按实施优先级来排，我建议：

### P0 现在就应该补规划

- MCP 能力层
- 记忆层概念
- 知识条目层概念
- 技能层概念

### P1 V1 可部分落地

- `knowledge_entry`
- `memory_entry`
- 驳回原因标签
- 复盘结论结构化

### P2 V1.1 再深入

- 轻量知识图谱
- 技能运行时
- RAG/检索增强正式上线
- 接受率/命中率反馈学习

## 10. 一句话结论

当前系统已经是一个很好的“AI 可接入型造价系统”，但如果要变成真正“AI 越用越强”的系统，还必须正式补上：

`MCP 能力层 + Skills 层 + System Memory 层 + Knowledge Layer + AI 增智闭环设计。`
