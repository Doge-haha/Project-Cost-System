这轮主要围绕“最近协作摘要”做连续增强，目标是让用户不必重新打开回流入口，也能直接在摘要区回看刚处理过的对象、当前来源页和最近一次交接对象。

### 这轮已经收口的能力

- 批量摘要不再只保留一句总述，支持在摘要区内默认折叠、按需展开“本轮对象”清单
- 展开后的对象清单会按对象类型分组，审核对象和过程单据对象可直接区分
- 每条对象支持 `复制入口`，复制后会记住本轮最近一次被复制的具体对象
- 再次进入相关页面时，如果存在最近复制对象，摘要会自动展开、定位并高亮那一条对象
- 用户可以 `清除最近复制对象`，只清掉单条高亮，不会把整轮批量摘要一起删掉
- 清除最近复制对象时，已经区分“系统自动展开”和“用户手动展开”，不会误收起用户主动展开的列表
- 对象清单的手动展开/收起偏好会随同一轮摘要一起记住，但不会误带到下一轮批量摘要
- 批量摘要和单条摘要都补上了“来源动作”，能直接显示 `回到待办页（待办）`、`回到审核处理页（待处理）`、`回到任务状态页（失败）` 这类语义化入口
- 当用户当前已经在来源页时，来源动作会显示 `已在此处`
- 非当前页场景下，已经把“来源动作”和“打开最近协作入口”做了去重，避免同时出现两个意义重复的跳转入口

- 风险：用户在任务状态页看到的是“失败条目子集”，但系统真正重试的可能是“成功入队事件里同类型的子集”，两者不是同一批对象

### 2. 子集重试会覆盖原始整批 payload

- 文件：[background-job-service.ts](/Users/huahaha/WorkSpace/something/新点SaaS计价/apps/api/src/modules/jobs/background-job-service.ts:406)
- 问题：当前重试会把过滤后的 `payload` 直接写回同一条 background job 记录
- 风险：一旦做过一次子集重试，原来的完整事件集就不在这条 job 记录里了，后面如果想再做整批重试，或改成另一个子集重试，可能已经没有完整上下文可用
- 明天接手时先确认：是否应该保留原始 payload，不要直接覆盖；或者把“本次重试范围”放到独立字段，而不是改写原始事件集

### 3. 前端重试范围和选中的任务可能错位

- 文件：[project-job-status-page.tsx](/Users/huahaha/WorkSpace/something/新点SaaS计价/apps/frontend/src/features/projects/project-job-status-page.tsx:1053)

- 明天接手时先确认：`selectedJob` 和 `selectedImportTask` 之间是不是应该建立强绑定，或者重试前至少要校验这两个对象来自同一条导入链

## 方案确认 - 2026-04-23 10:18

今天继续往下看时，先把“失败记录如何重建成可重试输入”这件事收成一个明确方案，避免后面又回到直接复用 `background_job.payload.events` 的歧路。

### 当前判断

- 上传导入链路里的失败条目来自 `import_task.metadata.failedItems`
- 原始入队 `background_job.payload.events` 只包含“通过首轮校验、被接受入队”的事件
- 所以“失败条目来源”和“原始 job payload 来源”不是同一集合，不能再直接把后者拿来做失败子集重试

### 先定下来的规则

1. 失败记录如果要支持重试，必须保留一份“可重建输入快照”
2. 这份快照和页面上的“预览字段”不是一回事
3. 子集重试未来应基于失败记录快照生成新的执行输入
4. 原始整批输入必须保持可追溯，不能在子集重试时被改写

### 下一轮真正实现时的建议落点

- 优先扩展 `failedItems`，或给它配套一个正式快照字段，而不是继续把 retry 逻辑绑在旧 job payload 上
- 如果失败记录字段不足以重建事件，就补“原始失败事件快照”而不是继续猜测
- 整批重试和子集重试要拆成两套输入来源：
  - 整批重试：原始整批输入
  - 子集重试：失败记录快照生成的新输入

### 当前状态说明

- 今天这一步还没有恢复“真实子集重试”
- 只是把后续实现边界在日志和 `OpenSpec` 里定清楚，避免再做出一个看上去能跑、但对象集合其实不一致的版本

### 2. 任务状态页修掉了一个真实的 URL 子筛选时序 bug

- 文件：[project-job-status-page.tsx](/Users/huahaha/WorkSpace/something/新点SaaS计价/apps/frontend/src/features/projects/project-job-status-page.tsx)
- 根因：页面在导入任务还没加载完成前，就会先校验 URL 里的 `failureResourceType` / `failureAction`
- 结果：本来有效的失败子筛选参数会在首屏初始化阶段被误删，导致用户通过深链接进入时，协作处理单元被错误降级成更粗粒度的失败原因视角
- 修法：只有等 `selectedImportTask` 加载完成后，才允许对这两个子筛选参数做有效性判断和自动清理

这条修复虽然是在 `jobs` 页里，但实际影响的是整条协作链里“失败子集协作单元能否稳定回放”的可信度，所以优先级比单纯的 UI 打磨更高。

### 3. 这条时序修复已经同步进规范

`openspec` 现在新增了这条边界：

- 带着 `failureReason`、`failureResourceType`、`failureAction` 进入任务状态页时，在导入任务还没加载完成前，不应先把这些子筛选误判为无效值并清掉
- 只有真正加载完成并确认这些参数不属于当前批次时，才允许自动清理

这样后面如果再有人调整任务状态页的 URL 清理逻辑，更容易意识到这里不是单纯的“脏参数清洗”，而是协作上下文保真问题。
