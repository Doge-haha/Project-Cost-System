# 工作日志 - 2026-04-23

## 交接记录 - 失败子集重试 review 跟进

### 背景

今天继续处理 `knowledge_extraction` 失败子集重试的 review 问题。review 重点是三类 P1 风险：

- 子集重试不能再过滤 `background_job.payload.events`，因为上传导入失败条目来自 `import_task.metadata.failedItems`，失败记录通常不在已入队的 `events` 列表里。
- 子集重试不能破坏原始整批 payload，否则后续整批重试或切换其他失败子集会失去原始输入。
- 前端不能把某个 import task 的失败筛选条件发给另一条 selected job。

### 当前已做

- `ImportFailedItem` 增加 `retryEventSnapshot`，上传解析失败/拒绝项会尽量保留原始输入快照；无法重建的解析错误仍为 `null`。
- `ImportTaskService.buildRetryEventsFromSnapshots()` 已基于 `metadata.failedItems` 和 `retryContext` 生成真正用于子集重试的 `retryEvents`，而不是从原 job `events` 里筛。
- `BackgroundJobProcessor` 已优先使用 `payload.retryEvents` 执行 `knowledge_extraction`，没有 `retryEvents` 时回落到原始 `events`。
- 前端 `project-job-status-page` 已增加 `selectedJobMatchesImportTask`，只有当前 job 与当前 import task 的 `latestJobId` 匹配时才会发送失败范围；不匹配时禁用重试按钮并提示切换导入批次。
- OpenSpec 已记录“失败子集重试必须保留可重建输入快照，不能破坏原始 payload”的要求。

### 刚刚补的关键修正

在 `apps/api/src/modules/jobs/background-job-service.ts` 的 `retryJob()` 里，生成下一次 payload 前会先剥离旧的派生字段：

- 删除上一轮残留的 `retryContext`
- 删除上一轮残留的 `retryEvents`
- 保留原始 `events` 和其他基础 payload 字段
- 只有本次请求带失败筛选范围时，才从 `failedItems.retryEventSnapshot` 重新构造新的 `retryEvents`

这个修正解决了一个隐蔽残留：如果做过一次子集重试，随后再点“整条任务重试”，旧的 `retryEvents` 不应继续影响本次执行。
