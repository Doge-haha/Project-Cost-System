## 一、代码 review 后的直接修复

### 1. 前端阻塞问题已清理

最开始先处理了 `apps/frontend` 的真实阻塞：

- `ProjectJobStatusPage` 的状态定义不一致，`uploadCompleted` 少字段，导致 `typecheck` 失败
- `retryJob()` 里存在失效变量引用，导致任务重试链不稳定
- 上传成功后的新批次选中逻辑不稳，导致“新旧批次对照提示”测试失败

这些问题已经修复，当前前端不再处于“红灯”状态。

### 2. 前端测试已补成回归护栏

今天不是只改实现，还把几类高风险场景都补成了回归测试：

- retry 接口只返回最小 job 字段时，页面仍保留当前导入批次上下文
- 上传新批次后，`queued / processing` 阶段不会过早展示对照结论
- 上传返回的新批次如果后续刷新结果中消失，页面会回退到旧失败批次，不进入悬空状态
- 当前 failure scope 只靠正式 `failureSnapshots` 也能启用 scoped retry

当前前端全量测试已通过。

## 二、API / Frontend / MCP Gateway 契约补强

### 1. API 已支持正式 `failureSnapshots`

今天把后端契约往前推了一步：

- `ImportTaskService.buildRetryEventsFromSnapshots()` 不再只依赖 `failedItems[*].retryEventSnapshot`
- 当失败条目没有内联快照时，会回退读取 `metadata.failureSnapshots`
- `GET /v1/projects/:projectId/import-tasks` 的测试也已明确校验 `metadata.failureSnapshots`

这意味着正式快照结构不再只是“前端准备好了”，而是后端真的能消费。

### 2. Frontend 已能只靠正式快照工作

前端这边也已经补上对应 UI 契约：

- 当 `failedItems` 里不再内联 `retryEventSnapshot`
- 但 `metadata.failureSnapshots` 里存在正式快照时
- 任务状态页依然会显示“可重建快照已就绪”
- 并允许 scoped retry

这一点已经由页面测试覆盖。

### 3. MCP Gateway 已具备可读 + 可操作入口

`apps/mcp-gateway` 今天补了两块关键能力：

- 新资源：`GET /v1/resources/import-failure-context`
- 新工具：`POST /v1/tools/retry-import-failure-scope`

当前资源会返回：

- 选中的 `importTaskId`
- 当前过滤后的 `failedItems`
- `failureSummary`
- `filteredSummary`
  - `itemCount`
  - `missingFieldCount`
  - `retrySnapshotCount`
  - 资源类型分布
  - 动作分布
- 正式 `failureSnapshots`

当前工具会把：

- `jobId`
- `failureReason`
- `failureResourceType`
- `failureAction`

透传到业务 API 的 `/v1/jobs/:jobId/retry`。

这意味着 AI / Agent 侧现在终于具备：

- 读当前 failure scope
- 判断当前范围是否可重试
- 基于当前范围直接发起 scoped retry

的完整入口。
