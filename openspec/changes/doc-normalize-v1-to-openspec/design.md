# Design: 当前架构现状描述

## 设计目标

本设计文档只描述仓库当前已经存在的架构现状和文档边界，不描述未来理想架构，也不作为新实现方案。

## 当前仓库现状

截至 2026-04-20，仓库采用单仓多目录组织，主要包含以下部分：

- `apps/api`：TypeScript 主业务后端，当前接口和测试覆盖最完整。
- `apps/worker`：TypeScript 异步任务进程，负责后台任务轮询、执行和 AI Runtime 调用。
- `apps/mcp-gateway`：TypeScript 的 MCP 能力入口，负责面向 AI Agent 的资源、工具和上下文聚合。
- `apps/ai-runtime`：Python AI Runtime，承接知识抽取、CLI 调用和 AI 相关运行时能力。
- `apps/frontend`：React + TypeScript 前端工作台，已存在基础骨架与测试文件。
- `legacy/backend-java`：冻结归档的 Java 原型，用于保留早期业务规则验证成果。
- `docs`：现有业务设计、架构、排期和导入文档集合。

## 当前架构判断

当前仓库已经不再以 `legacy/backend-java` 作为主线实现，而是以 `apps/api + apps/worker + apps/mcp-gateway + apps/ai-runtime + apps/frontend` 作为现行主目录结构。

同时，文档层存在两类内容并存：

- 一类是早期面向“目标实现”的技术方案文档。
- 一类是已经落到当前仓库结构和模块边界的执行基线文档。

因此，本次规范化需要区分：

- “当前现状”是什么。
- “历史方案”与“未来方向”是什么。

## 当前业务与架构文档边界

现有文档体系大致分为四层：

- 产品与业务设计层：以 `设计文档_v1.0_优化中.md` 为总基线。
- 架构与规则层：包括数据模型、状态机、权限矩阵、技术架构、后端重构方案等。
- 实施与排期层：包括迭代拆分、执行计划和路线图。
- 派工与导入层：包括 Jira 卡片、CSV 模板和导入检查清单。

这意味着当前最缺的不是“更多新内容”，而是“更清晰的规范化归档方式”，让稳定业务规则、当前架构现状和后续执行任务分层管理。

## 本次 OpenSpec 拆分策略

本次只建立最小骨架，避免一次性大拆文档：

- `specs`：只先抽取稳定、跨实现都成立的“9 阶段业务模型”。
- `design`：只记录当前仓库现状，不对未来实现作承诺。
- `tasks`：只记录文档整理动作，不派生代码任务。

## 风险与约束

- 当前仓库中同时存在“历史 Java 方案”和“现行 TypeScript/Python 主线”，整理时必须避免混写。
- 架构文档中有部分“推荐方案”措辞，本次 design 不应直接复述为“已实现事实”。
- 原始设计文档章节很多，本次只做第一步规范化，不追求一次拆完整本大文档。
