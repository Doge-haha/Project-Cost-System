# Spec: 9阶段业务模型

## 概述

本 spec 定义新点 SaaS 造价系统的统一业务阶段模型，用于约束项目全生命周期中的阶段名称、顺序语义、边界和配置口径。

本 spec 来源于 [设计文档_v1.0_优化中.md](/Users/huahaha/WorkSpace/something/新点SaaS计价/设计文档_v1.0_优化中.md) 第一章内容，目标是把稳定业务规则从大文档中抽离出来。

## ADDED Requirements

### Requirement: MUST 系统必须采用统一的九阶段业务模型

MUST 系统必须以统一的九阶段业务模型描述项目全生命周期，阶段依次为：

1. 投资估算
2. 目标成本
3. 招标清单
4. 招标控制价
5. 投标报价
6. 合同清单
7. 施工过程
8. 竣工结算
9. 项目复盘

#### Scenario: 读取全生命周期阶段定义

- **GIVEN** 使用者查看系统业务阶段定义
- **WHEN** 系统展示标准生命周期
- **THEN** 系统应展示以上九个阶段
- **AND** 顺序应与标准全链路保持一致

### Requirement: MUST 每个阶段必须具有明确业务语义

MUST 系统必须为每个阶段定义唯一的业务目的、核心输入、核心操作、主要产出和下游流转关系。

#### Scenario: 识别阶段职责

- **GIVEN** 用户查看某一业务阶段
- **WHEN** 用户查询该阶段定义
- **THEN** 用户应能看到该阶段负责解决的问题、依赖的输入和产生的输出
- **AND** 该定义不应与其他阶段混用

### Requirement: MUST 阶段、子工作和模块必须区分

MUST 系统文档必须区分以下概念：

- `阶段`：项目全生命周期中的一级业务环节。
- `子工作`：某阶段内部可并行开展的具体业务活动。
- `模块`：支撑一个或多个阶段处理的系统功能单元。

#### Scenario: 描述施工过程

- **GIVEN** 文档描述“施工过程”
- **WHEN** 文档说明其组成内容
- **THEN** “施工过程”应被定义为一级阶段
- **AND** “设计变更、现场签证、进度款申报、材料认价、工程计量、变更结算、其他事项”应被定义为该阶段下的子工作

### Requirement: MUST 项目必须允许按模板启用阶段

MUST 系统必须支持按项目选择启用阶段，并允许基于模板或自定义方式配置阶段集合。

#### Scenario: 基于模板创建项目

- **GIVEN** 用户创建项目
- **WHEN** 用户选择内置模板或自定义模板
- **THEN** 系统应允许基于九阶段模型启用部分阶段
- **AND** 未启用阶段不进入该项目的业务链路

### Requirement: MUST 启用阶段必须具备独立配置能力

MUST 对于项目中启用的每个阶段，系统必须支持独立配置顺序、负责人、权限范围、完成动作和 AI 辅助开关。

#### Scenario: 配置项目阶段

- **GIVEN** 某项目启用了多个业务阶段
- **WHEN** 项目管理员调整阶段配置
- **THEN** 系统应允许按阶段分别配置负责人和权限
- **AND** 系统应允许按阶段控制是否启用 AI 辅助

### Requirement: MUST 阶段顺序调整不得破坏全链路约束

MUST 系统可以允许项目在九阶段模型内进行适配性调整，但不得形成环形流转，也不得混淆上下游关系。

#### Scenario: 调整项目阶段顺序

- **GIVEN** 用户希望按项目实际情况调整已启用阶段顺序
- **WHEN** 用户保存阶段流转配置
- **THEN** 系统应允许跳过不适用阶段
- **AND** 已启用阶段之间必须仍能形成清晰的上下游链路
- **AND** 不允许出现环形流转
