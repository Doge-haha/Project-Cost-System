# Spec: 导入任务专题

## 概述

本 spec 定义新点 SaaS 造价系统中的导入任务专题规则，用于统一导入入口、导入任务状态、错误报告、文件存储边界和与业务对象生成的关系。

## ADDED Requirements

### Requirement: 正式导入必须通过独立导入任务承接

批量导入不得直接在页面中无痕落库，而应通过明确的导入任务对象承接文件、状态和结果。

#### Scenario: 发起批量导入

- **GIVEN** 用户上传 Excel、XML、GBQ 或其他导入文件
- **WHEN** 系统受理请求
- **THEN** 应创建独立导入任务

### Requirement: 导入任务必须具备独立状态反馈

导入任务必须至少支持排队、处理中、成功、部分成功和失败等结果语义，以便用户理解导入结果。

#### Scenario: 查看导入进度

- **GIVEN** 用户已提交导入任务
- **WHEN** 用户查看任务状态
- **THEN** 系统应返回明确的处理状态和结果摘要

#### Scenario: 导入任务返回正式稳定字段

- **GIVEN** 用户在项目内查看某个导入任务
- **WHEN** 系统返回任务详情
- **THEN** 应稳定返回来源文件名、批次号、失败明细、重试次数、重试上限和是否可继续重试

### Requirement: 导入失败必须保留明细并支持错误报告

导入失败或部分成功时，系统必须保留失败明细，并允许用户下载错误报告或查看错误列表。

#### Scenario: 导入出现错误

- **GIVEN** 导入任务存在失败记录
- **WHEN** 用户查看结果
- **THEN** 系统应提供失败明细或错误报告能力

#### Scenario: 部分成功时保留失败条目明细

- **GIVEN** 某批导入只有部分条目通过校验
- **WHEN** 用户查看批次详情
- **THEN** 系统应保留失败条目列表，并区分每条失败的行号、原因类别、错误信息和基础上下文

#### Scenario: 失败原因按类别归类展示

- **GIVEN** 某批导入包含多种失败原因
- **WHEN** 用户查看批次详情
- **THEN** 系统应提供按原因类别聚合的失败摘要，帮助用户优先处理同类问题

#### Scenario: 用户可按失败原因筛选条目

- **GIVEN** 某批导入已经返回失败原因归类和失败条目明细
- **WHEN** 用户在任务状态页选择某个失败原因
- **THEN** 系统应仅展示该原因对应的失败条目，并明确当前筛选视角

### Requirement: 导入文件必须与业务元数据分离存储

导入文件实体不应直接塞入业务主表，正式文件应走对象存储或独立文件存储，业务系统只保留元数据和任务关联。

#### Scenario: 保存导入文件

- **GIVEN** 用户上传导入文件
- **WHEN** 系统持久化导入结果
- **THEN** 文件实体应与业务主数据分离存储

### Requirement: 导入成功后必须生成明确业务结果

不同类型导入成功后，系统必须明确生成初始版本、明细对象或标准映射结果，不能停留在“文件已上传”层面。

#### Scenario: 导入清单成功

- **GIVEN** 用户导入清单文件
- **WHEN** 导入任务成功完成
- **THEN** 系统应生成对应业务版本或明细结果

### Requirement: 上传导入必须支持真实文件批次与解析摘要

正式导入不应只停留在“任务已创建”，而应支持真实文件上传、格式识别、解析概览和条目预览，帮助用户判断批次结构是否正确。

#### Scenario: 上传结构化文本文件创建导入批次

- **GIVEN** 用户上传 JSON 数组、JSONL 或 NDJSON 文本文件
- **WHEN** 系统受理上传
- **THEN** 应自动解析内容、生成新的导入批次并排入异步处理

#### Scenario: 批次详情展示解析摘要与预览

- **GIVEN** 某批导入来自真实文件上传
- **WHEN** 用户展开批次详情
- **THEN** 系统应展示解析概览、条目预览和重试历史，帮助用户理解这批数据的结构和处理轨迹

### Requirement: 导入运营视角必须支持协作上下文

当用户围绕某类失败原因开展排查、重试或协作时，系统应允许保留并传播同一协作视角，而不是每次重新筛选。

#### Scenario: 通过协作视角返回失败原因筛选

- **GIVEN** 用户已经围绕某个失败原因查看批次详情
- **WHEN** 用户返回任务状态页、待办页或项目工作台
- **THEN** 系统应能保留同一失败原因视角，帮助用户连续处理

#### Scenario: 复制协作链接分享当前问题视角

- **GIVEN** 用户已经选定某个失败原因视角
- **WHEN** 用户复制当前链接并分享给他人
- **THEN** 接收方打开后应直接落在相同批次或页面视角，而不需要重新筛选

#### Scenario: 在失败原因视角下定位到单条失败记录

- **GIVEN** 用户已经围绕某个失败原因查看失败条目列表
- **WHEN** 用户继续选择其中某一条失败记录
- **THEN** 系统应支持进入单条回看视角，并明确展示该条记录的行号、原因、错误信息和基础上下文

#### Scenario: 单条失败记录定位可通过链接继续协作

- **GIVEN** 用户已经进入某条失败记录的单条回看视角
- **WHEN** 用户复制当前链接并分享给协作同事
- **THEN** 接收方打开后应直接定位到同一条失败记录，而不需要再次在列表中查找

#### Scenario: 单条失败记录应支持逐条切换回看

- **GIVEN** 用户已经进入某个失败原因下的单条回看视角
- **WHEN** 用户继续查看上一条或下一条失败记录
- **THEN** 系统应在保留当前筛选上下文的前提下切换到相邻条目，并明确当前所处序号

#### Scenario: 单条失败记录应展示原始预览与字段差异

- **GIVEN** 某条失败记录同时保留了原始条目预览
- **WHEN** 用户进入单条回看视角
- **THEN** 系统应展示该条记录的原始字段、失败条目字段以及可直接帮助排查的缺失字段或差异字段

#### Scenario: 当前失败原因视角应展示轻量运营摘要

- **GIVEN** 用户已经进入某个失败原因视角或全部失败条目视角
- **WHEN** 页面展示当前筛选结果
- **THEN** 系统应汇总当前范围内的资源类型分布、动作分布、缺字段条目数以及可回看原始预览的条目数，帮助用户快速判断问题形态

#### Scenario: 用户可从运营摘要直接收束到子集

- **GIVEN** 当前失败原因视角已经展示资源类型分布或动作分布
- **WHEN** 用户点击某个资源类型或动作项
- **THEN** 系统应直接把失败条目列表收束到对应子集，并通过 URL 保留该子集视角，方便继续协作或回流

#### Scenario: 当前子集导出应与收束视角一致

- **GIVEN** 用户已经通过资源类型或动作把当前失败原因视角继续收束到更小子集
- **WHEN** 用户发起当前子集导出
- **THEN** 导出结果应与当前子集视角保持一致，而不是回退到更大的失败原因范围

#### Scenario: 当前子集应给出批量处理建议

- **GIVEN** 页面已经拿到当前失败原因或当前子集的统计结果
- **WHEN** 用户查看该范围的运营摘要
- **THEN** 系统应给出只读的批量处理建议，例如优先回源补字段、优先检查映射规则、优先逐条复核或优先导出当前范围交给上游排查

#### Scenario: 当前子集应支持复制处理摘要

- **GIVEN** 用户已经完成对当前失败原因或当前子集的判断
- **WHEN** 用户复制当前子集处理摘要
- **THEN** 系统应生成一段可直接发给协作同事或上游的数据排查文案，其中包含当前范围、条目数、缺字段数量、预览覆盖情况和建议动作

#### Scenario: 当前子集应支持按协作对象生成不同摘要

- **GIVEN** 用户需要把当前范围交接给不同对象
- **WHEN** 用户选择复制给协作同事或复制给上游数据方
- **THEN** 系统应复用同一份当前范围统计，但按接收对象生成不同重点的摘要文案

#### Scenario: 协作文案应附带当前视角链接

- **GIVEN** 用户已经生成面向协作同事或上游数据方的摘要文案
- **WHEN** 用户执行复制
- **THEN** 文案中应同时附带当前页面链接，让接收方可以直接打开同一失败原因或子集视角

#### Scenario: 上游数据方文案应提示对应导出文件

- **GIVEN** 用户准备把当前范围交给上游数据方排查
- **WHEN** 用户复制上游数据方版摘要
- **THEN** 文案中应提示建议对应的导出文件名，帮助上游把摘要、链接和导出材料对应起来

#### Scenario: 当前失败子集可生成结构化处理单

- **GIVEN** 用户已经围绕某个失败原因继续收束到资源类型、动作或其他更小的失败子集
- **WHEN** 用户需要把这组问题作为一个独立处理单元转交给协作同事或上游数据方
- **THEN** 系统应基于当前子集直接生成结构化处理单，并至少包含处理范围、失败条目数、主要资源类型、主要动作、建议动作、建议导出文件名和当前视角链接，减少再次口头解释当前范围的成本

#### Scenario: 当前失败子集可沉淀为失败记录对象清单

- **GIVEN** 用户已经围绕某个失败原因继续收束到更小的失败子集，并在其中查看或复制过具体失败记录
- **WHEN** 系统保存最近一次失败子集协作记录
- **THEN** 系统应支持把当前子集中的失败记录按对象清单方式沉淀下来，并至少保留每条记录的行号标签、可回看的处理路径以及最近一次被复制的具体记录，便于用户在入口页重新展开、定位和继续交接

### Requirement: 可重试失败记录必须保留可重建输入快照

如果系统希望支持基于失败记录或失败子集重新入队，就不能只保留错误摘要，而必须保留足以重新生成执行输入的结构化快照。

#### Scenario: 失败记录保留可重建输入

- **GIVEN** 某条导入记录在校验或解析阶段失败，未进入正常异步处理链
- **WHEN** 系统把该条记录写入 `failedItems` 或等价失败明细集合
- **THEN** 系统应同时保留该条记录的可重建输入快照，至少保证后续可以在不依赖原始 `background_job.payload.events` 的前提下重建出新的执行输入

#### Scenario: 原始预览与可重试快照应区分职责

- **GIVEN** 系统已经保存失败记录的原始预览字段和用于重试的结构化快照
- **WHEN** 用户查看失败记录详情或后续触发重试
- **THEN** 页面展示可继续使用预览字段帮助排查，但后台生成重试输入时应以可重试快照为准，避免把展示用途的预览字段误当成执行输入

### Requirement: 子集重试不得改写原始整批输入

如果系统支持围绕失败子集重试，子集重试应被视为一次新的执行请求，而不是对原始整批输入的破坏性更新。

#### Scenario: 子集重试生成新的执行输入

- **GIVEN** 用户围绕某个失败原因继续收束到资源类型、动作或更小的失败子集
- **WHEN** 用户发起该子集的重试
- **THEN** 系统应基于失败记录快照生成新的执行输入，并允许保留本次重试范围上下文；原始整批输入应继续保持可追溯、可再次整批重试

#### Scenario: 原始整批输入在子集重试后仍可恢复

- **GIVEN** 某批导入已经做过一次或多次子集重试
- **WHEN** 用户后续选择重新发起整批重试，或切换到另一组失败子集再试
- **THEN** 系统仍应能够恢复原始整批输入或从正式来源重新构建整批输入，而不应因为之前的子集重试而永久丢失整批上下文

## Implementation Notes

以下内容是当前仓库实现现状的观察，不属于规范要求本身：

- 当前 TypeScript API 已补充正式 `import_task` 模块，并通过 `GET /v1/projects/:projectId/import-tasks`、`POST /v1/projects/:projectId/import-tasks` 与 `POST /v1/projects/:projectId/import-tasks/upload` 提供项目内导入任务能力。
- 当前上传导入已支持 JSON 数组、JSONL 和 NDJSON 文本文件，并能自动生成导入批次。
- 当前导入任务已稳定返回 `sourceFileName`、`sourceBatchNo`、`failureDetails`、`retryCount`、`retryLimit` 和 `canRetry` 等字段。
- 当前导入批次详情已补充解析概览、条目预览和重试历史，用于支撑首版导入运营视角。
- 当前导入链路已支持“合法条目继续入队、非法条目沉淀失败明细”的处理方式，并把失败条目明细与失败原因归类保存在任务元数据中。
- 当前 TypeScript API 已补充 `GET /v1/projects/:projectId/import-tasks/:taskId/error-report`，支持把失败条目明细下载为错误报告。
- 当前前端已支持在任务状态页中按失败原因筛选失败条目，并围绕该视角复制协作链接、重试失败任务和回流到待办页或项目工作台。
- 当前前端已支持在任务状态页中继续定位单条失败记录，并通过 URL 保留 `failedLine` 深链视角，便于协作同事直接回看到同一条问题数据。
- 当前前端已支持在单条回看视角中继续切换上一条或下一条失败记录，并显示当前位于筛选结果中的第几条，方便运营同学逐条排查。
- 当前前端已在单条回看视角中补充原始条目预览和字段交叉检查，帮助用户直接判断是缺字段、字段值不对，还是原始数据本身就不完整。
- 当前前端已在失败原因视角中补充轻量运营摘要，用于先判断这批问题是否集中在同一种资源、动作或缺字段模式，再决定是否逐条排查。
- 当前前端已支持从运营摘要直接点击某个资源类型或动作，把当前原因视角继续收束到更小的失败条目子集。
- 当前前端已支持在存在资源类型或动作二级收束时，直接导出当前子集，并让导出文件名带上当前子集范围标识。
- 当前前端已在当前范围旁边补充只读的批量动作建议区，用于把统计结果直接转成下一步处理建议，而不是要求用户自己解释统计数字。
- 当前前端已支持直接复制当前子集处理摘要，便于把判断结果以文本形式交接给协作同事或上游数据提供方。
- 当前前端已把处理摘要拆成“协作同事版”和“上游数据方版”两种复制出口，方便按对象发送不同重点的交接文案。
- 当前前端在两类处理摘要中都附带了当前页面链接，方便接收方直接打开同一筛选视角继续处理。
- 当前前端会在上游数据方版摘要中附带建议导出文件名提示，帮助上游快速对应到应查看的导出材料。
- 当前前端已支持在失败原因或当前子集视角中生成结构化“失败子集处理单”，把当前范围、条目数、建议动作、建议导出文件名和处理链接收束成一个可直接转交的协作单元。
- 当前前端已支持把失败子集中的具体失败记录沉淀为最近协作对象清单，并允许在入口页继续展开、定位最近复制记录或重新复制单条失败记录入口。
- 当前前端已支持在任务状态页直接下载当前导入批次的错误报告，并显式区分“导出当前筛选”和“导出整批”；导出文件名也会带上范围标识与失败原因码，便于协作排查和线下转交；若当前筛选下没有失败条目，则前端会直接跳过空导出并提示用户。
- 当前实现已补充 `retryContext` 记录失败子集重试范围，并通过 `failedItems.retryEventSnapshot` 保存失败记录的可重建输入快照；该字段是失败子集重试的正式输入来源，不应与页面展示用的原始预览字段混用。
- 当前实现会在本次子集重试时把 `failedItems.retryEventSnapshot` 派生成 `background_job.payload.retryEvents`，`retryEvents` 只代表本次重试的执行输入，不是原始整批输入；后续整批重试必须清理旧 `retryEvents` 并回到原始 `events`。
- 当前实现要求选定失败范围内的失败条目全部具备 `retryEventSnapshot` 后才允许作为范围重试输入；如果只有部分条目可重建，后端会拒绝该范围重试，前端也会降级为整条任务重试提示，避免静默漏掉不可重建条目。
- 当前实现由 `BackgroundJobProcessor` 在执行前统一把 `retryEvents` 归一到 `events`，业务处理服务只消费归一后的 `events`，避免多个执行入口重复决定本次实际处理范围。
- 当前前端已识别 `IMPORT_TASK_RETRY_INPUT_INCOMPLETE` 错误码，并在任务状态页提示用户先导出当前范围或回源修数后再重新导入，而不是只展示通用失败提示。
- 当前前端已对协作视角提供无效值自动清理、手动清除和提示自动消失等体验收口。
- 现有文档仍建议导入文件实体与业务主数据分离存储，实体文件建议后续走对象存储，业务系统保留任务和结果摘要。
