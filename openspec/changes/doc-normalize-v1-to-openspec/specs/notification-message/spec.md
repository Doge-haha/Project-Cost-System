# Spec: 通知与消息专题

## 概述

本 spec 定义新点 SaaS 造价系统中的通知与消息专题规则，用于统一待审核提醒、批注提醒、阶段流转通知、消息入口和消息摘要边界。

## ADDED Requirements

### Requirement: 系统必须提供统一消息入口

系统必须提供统一消息入口或通知中心，用于承载待审核项、批注提醒和流程变更通知。

#### Scenario: 打开通知中心

- **GIVEN** 用户位于系统全局导航中
- **WHEN** 用户打开通知中心
- **THEN** 应能查看统一消息列表

### Requirement: 审核与流程动作必须触发消息提醒

提交审核、审核通过、审核驳回、阶段完成和跨阶段流转等关键流程动作必须触发消息提醒。

#### Scenario: 提交流转动作

- **GIVEN** 某关键流程动作成功完成
- **WHEN** 系统处理后续联动
- **THEN** 应向相关责任人生成通知消息

### Requirement: 批注和协作反馈必须可进入消息列表

批注提醒、协作反馈和需要关注的风险项必须能够进入统一消息列表，而不是散落在业务页面中无法集中查看。

#### Scenario: 产生协作提醒

- **GIVEN** 用户收到批注或协作反馈
- **WHEN** 系统生成提醒
- **THEN** 该提醒应可在统一消息列表中查看

### Requirement: 消息必须关联明确业务上下文

每条正式消息都必须关联明确的项目、阶段、资源或动作来源，以便用户从消息直接定位到业务对象。

#### Scenario: 查看消息详情

- **GIVEN** 用户打开某条消息
- **WHEN** 系统展示消息内容
- **THEN** 应能说明其关联项目、阶段或资源来源

### Requirement: 工作台和全局导航必须展示消息摘要

通知与消息不仅应在通知中心可见，也应在项目工作台或全局导航中展示未读数或摘要提示。

#### Scenario: 查看消息摘要

- **GIVEN** 用户进入系统首页或项目工作台
- **WHEN** 系统展示导航摘要
- **THEN** 应能看到消息入口及未读或待处理提示

## Implementation Notes

以下内容是当前仓库实现现状的观察，不属于规范要求本身：

- 当前设计文档已明确全局导航中存在“通知中心”，并将待审核项、批注提醒等作为主要消息类型。
- 当前仓库中未见独立通知服务、通知表或消息路由实现，说明通知专题目前主要停留在产品与架构设计层。
- 工作台右侧侧栏和全局导航入口已经为消息聚合预留了承载位，后续实现可以直接挂接到这些稳定入口上。

这些现状说明通知与消息边界已经明确，但正式消息系统仍属于后续实现内容。
