# Spec: 权限边界

## 概述

本 spec 定义新点 SaaS 造价系统的权限边界规则，用于约束平台角色、项目成员、阶段范围、专业范围、锁定优先级和审编分离等稳定业务规则。

## ADDED Requirements

### Requirement: 权限判断必须采用多层控制模型

系统必须以多层权限模型进行访问控制，至少同时考虑以下维度：

1. 平台角色
2. 项目成员关系
3. 阶段范围
4. 专业或单体范围
5. 资源当前状态

#### Scenario: 判断用户能否访问项目资源

- **GIVEN** 用户尝试访问某个项目内资源
- **WHEN** 系统执行权限判断
- **THEN** 系统必须先确认该用户已进入该项目的成员上下文
- **AND** 系统必须继续结合阶段、范围和资源状态判断最终是否允许访问

### Requirement: 平台角色不能替代项目成员授权

系统角色用于定义平台级能力边界，但除系统管理员等特权角色外，平台角色本身不能直接替代项目成员授权。

#### Scenario: 非系统管理员直接访问项目

- **GIVEN** 一个拥有平台角色的用户
- **WHEN** 该用户不是目标项目成员
- **THEN** 系统不应仅凭平台角色授予项目内访问权

### Requirement: 系统必须区分角色与业务身份

系统文档必须区分“平台角色”和“业务身份”：

- 平台角色决定用户在平台层面的模块进入能力。
- 业务身份决定用户在特定业务场景和阶段中的职责语义。

#### Scenario: 同一角色跨阶段参与业务

- **GIVEN** 某用户的平台角色为造价员
- **WHEN** 该用户在不同项目阶段承担不同业务职责
- **THEN** 系统文档应允许其被标识为不同业务身份
- **AND** 业务身份不能单独脱离项目成员关系生效

### Requirement: 启用阶段的权限必须按阶段独立裁剪

对于已启用的项目阶段，系统必须支持按阶段裁剪查看、编辑、提交和审核权限。

#### Scenario: 用户访问非授权阶段

- **GIVEN** 用户属于某项目成员
- **WHEN** 用户尝试访问未被授权的阶段资源
- **THEN** 系统应拒绝其访问或编辑该阶段上下文

### Requirement: 专业与单体范围必须限制实际可操作边界

即使用户具备项目成员身份和阶段权限，其实际可操作数据范围仍必须受专业范围和单体范围限制。

#### Scenario: 限定专业范围

- **GIVEN** 用户在项目中被授予特定专业范围
- **WHEN** 用户尝试访问未授权专业的数据
- **THEN** 系统应拒绝该访问或编辑动作

### Requirement: 审核权限与编制权限必须隔离

对于同一资源对象，系统必须遵守审编分离原则。

#### Scenario: 提交人与审核人冲突

- **GIVEN** 某用户已经作为提交人提交资源进入审核
- **WHEN** 该用户尝试对同一资源执行审核动作
- **THEN** 系统必须拒绝该审核动作

#### Scenario: 同一用户同时拥有提交与审核能力

- **GIVEN** 某阶段资源需要编制和审核
- **WHEN** 系统为同一资源判断操作权限
- **THEN** 不应允许同一用户同时作为该资源的编制执行者和审核执行者

### Requirement: 锁定状态优先级必须高于编辑权限

资源一旦进入锁定态、基线态或其他禁止直接修改的状态，状态约束必须高于普通编辑权限。

#### Scenario: 有编辑权限但资源已锁定

- **GIVEN** 用户原本具备该资源的编辑权限
- **WHEN** 该资源已经被锁定
- **THEN** 系统不应允许直接修改该资源
- **AND** 如需继续变更，应通过后续业务单据或解锁流程处理

### Requirement: 高敏感阶段必须支持额外可见性限制

对于招标控制价等高敏感阶段，系统必须支持在一般权限模型之外施加更严格的可见性控制。

#### Scenario: 未开标前访问招标控制价

- **GIVEN** 项目处于招标控制价敏感期
- **WHEN** 非指定负责人或非指定审核人尝试查看该阶段数据
- **THEN** 系统应限制其可见性

### Requirement: 项目负责人应具备跨阶段总览能力

项目负责人必须能够在项目范围内查看跨阶段总体视图，以支持协调、配置和总览管理。

#### Scenario: 查看全项目汇总

- **GIVEN** 某用户是项目负责人
- **WHEN** 其访问项目级汇总或跨阶段总览
- **THEN** 系统应允许其查看项目范围内的总体视图

### Requirement: 系统管理员必须保留平台级特权能力

系统管理员必须保留平台级管理和必要的权限绕过能力，用于用户管理、权限配置和平台维护。

#### Scenario: 系统级管理操作

- **GIVEN** 某用户是系统管理员
- **WHEN** 其执行用户管理、权限配置或平台运维相关操作
- **THEN** 系统应允许其执行对应的平台级动作

## Implementation Notes

以下内容是当前仓库实现现状的观察，不属于规范要求本身：

- `apps/api` 当前通过 `ProjectAuthorizationService` 进行项目内 `view/edit` 判断。
- 当前代码中的角色策略相对简化，主要覆盖 `system_admin`、`project_owner`、`cost_engineer`、`reviewer`。
- 当前实现已存在基于 `stage` 与 `discipline` 的 scope 检查。
- `apps/mcp-gateway` 当前对部分写工具和知识预览工具做了角色级裁剪。

这些现状说明当前实现已经具备权限模型骨架，但不应反向限制本 spec 的业务表达范围。
