# Spec: 计价引擎专题

## 概述

本 spec 定义新点 SaaS 造价系统中的计价引擎专题规则，用于统一综合单价计算、费用分摊、重算触发、范围控制和人工调价审计边界。

## ADDED Requirements

### Requirement: 计价引擎必须以清单、定额、价目和取费为统一输入

正式计价结果必须基于清单、关联定额、所选价目版本和取费规则共同计算，不能只依赖单一来源得出最终金额。

#### Scenario: 执行单项计价

- **GIVEN** 某清单项已具备工程数量和定额明细
- **WHEN** 系统执行计价
- **THEN** 应同时使用清单、定额、价目和取费作为输入

### Requirement: 计价结果必须区分系统值、人工值和最终生效值

计价模块必须显式区分系统计算值、人工调整值和最终对外使用值，避免人工修改直接覆盖系统基线。

#### Scenario: 查看计价结果

- **GIVEN** 某清单项已完成计价
- **WHEN** 用户查看结果明细
- **THEN** 系统应能区分系统值、人工值和最终生效值

### Requirement: 分摊策略必须支持多层级范围

措施费、规费和税金的分摊策略必须至少支持项目、专业、单体和清单等层级范围，以适配不同业务场景。

#### Scenario: 配置分摊策略

- **GIVEN** 项目存在不同计价口径
- **WHEN** 项目选择分摊范围
- **THEN** 系统应支持按项目、专业、单体或清单进行分摊

### Requirement: 重算必须由价格相关变动自动触发

凡是影响结果金额的关键变动都必须触发重算或进入待重算状态，不得在正式结果中长期保留失效计算值。

#### Scenario: 修改影响价格字段

- **GIVEN** 用户修改工程量、定额明细、含量系数或取费参数
- **WHEN** 修改被保存
- **THEN** 系统应触发相应重算或标记受影响范围待重算

### Requirement: 重算范围必须与触发来源匹配

重算不应无差别全量执行，而应根据触发来源控制在单项、局部汇总或项目范围内。

#### Scenario: 执行重算

- **GIVEN** 系统识别到某次变动来源
- **WHEN** 确定重算范围
- **THEN** 应根据单项编辑、定额变更、取费调整或价目切换选择对应范围

#### Scenario: 项目级重算在入队前校验作用范围

- **GIVEN** 用户请求按项目、阶段或专业范围发起重算
- **WHEN** 系统准备创建项目级重算任务
- **THEN** 应先确认目标范围内确实存在可重算版本
- **AND** 当目标范围为空时不应创建空任务

#### Scenario: 项目级重算校验价目和取费配置兼容性

- **GIVEN** 用户在项目级重算中指定价目版本或取费模板
- **WHEN** 系统准备创建重算任务
- **THEN** 应先校验价目版本专业是否与目标范围兼容
- **AND** 应先校验取费模板阶段范围是否与目标范围兼容

#### Scenario: 版本级重算校验价目和取费配置兼容性

- **GIVEN** 用户在版本级重算中指定价目版本或取费模板
- **WHEN** 系统执行同步重算
- **THEN** 应先校验价目版本专业是否与该版本兼容
- **AND** 应先校验取费模板阶段范围是否与该版本兼容

### Requirement: 取费规则命中必须按明确优先级生效

当同一类取费存在专业专属规则和通用规则时，系统不得简单叠加同类规则，而应按明确优先级选择最终生效规则。

#### Scenario: 同类取费规则同时存在专属和通用口径

- **GIVEN** 某取费模板下同一 `feeType` 同时存在专业专属规则和通用规则
- **WHEN** 系统为某个专业执行计价
- **THEN** 应优先命中专业专属规则
- **AND** 通用规则只应作为兜底规则而不是与专属规则重复叠加

### Requirement: 人工调价必须保留原因和审计链路

允许人工调价的阶段中，人工调价必须记录调整原因，并保留系统基线和操作审计链路。

#### Scenario: 提交人工调价

- **GIVEN** 当前阶段允许人工调价
- **WHEN** 用户保存人工调整结果
- **THEN** 系统应记录调整原因
- **AND** 不得覆盖原系统计算值

### Requirement: 缺少必要计价输入时不得输出正式结果

当缺少有效定额、匹配价目或取费规则时，系统不得伪造完整正式结果，而应返回校验问题或阻止正式计价。

#### Scenario: 输入条件不完整

- **GIVEN** 清单项缺少必要计价输入
- **WHEN** 系统尝试计算
- **THEN** 应提示缺失项并拒绝输出完整正式计价结果

### Requirement: 批量重算必须返回可解释的跳过结果

批量重算不应因为单个异常条目直接中断整批处理，而应在可容忍场景下继续处理其余条目，并把跳过条目与原因明确返回。

#### Scenario: 版本级重算跳过空定额清单项

- **GIVEN** 某清单项没有任何定额明细
- **WHEN** 用户执行版本级重算
- **THEN** 系统应跳过该清单项
- **AND** 在结果中返回该条目的跳过原因与基础细节

#### Scenario: 版本级重算跳过非法工程量清单项

- **GIVEN** 某清单项工程量小于等于零
- **WHEN** 用户执行版本级重算
- **THEN** 系统应跳过该清单项
- **AND** 在结果中返回非法工程量原因与对应数量

#### Scenario: 版本级重算跳过价目匹配失败清单项

- **GIVEN** 某清单项存在定额明细但无法在当前价目版本中完成全部匹配
- **WHEN** 用户执行版本级重算
- **THEN** 系统应跳过该清单项而不是中断整批重算
- **AND** 在结果中返回未匹配的定额编码列表

#### Scenario: 项目级重算聚合所有跳过条目

- **GIVEN** 项目级重算覆盖多个版本
- **WHEN** 系统完成重算
- **THEN** 最终结果应聚合所有版本的跳过条目
- **AND** 每条记录应明确所属版本、清单项和跳过原因

## Implementation Notes

以下内容是当前仓库实现现状的观察，不属于规范要求本身：

- `apps/api/src/modules/engine/calculate-service.ts` 当前已具备按清单项、价目版本和取费模板进行基础计算的服务骨架。
- 当前实现已写入 `systemUnitPrice`、`finalUnitPrice`、`systemAmount`、`finalAmount` 和 `calculatedAt` 等结果字段。
- 当前仓库测试中已覆盖 `/v1/engine/calculate`、版本级 `recalculate` 和项目级 `recalculate` 主线。
- 当前实现已支持在批量重算中返回 `skippedItems`，首版覆盖 `missing_quota_lines`、`invalid_quantity` 和 `unmatched_price_items` 等原因，并带基础排查细节。
- 当前实现已在项目级重算入队前补充范围预校验，能够拒绝空目标范围、价目版本专业不匹配和取费模板阶段范围不匹配。
- 当前实现已在版本级重算中补充价目版本与取费模板的前置兼容性校验。
- 当前实现已在取费规则命中时支持同类规则的优先级选择，避免专业专属规则与通用规则重复叠加。
- 完整分摊策略、更细粒度重算控制和更丰富的跳过原因分类仍属于后续实现。

这些现状说明计价主链已经从“最小可算”推进到“最小可控、可解释”的阶段，但完整分摊与高级策略仍高于当前实现范围。
