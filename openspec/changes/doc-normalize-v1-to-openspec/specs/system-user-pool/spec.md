# Spec: 系统用户池专题

## 概述

本 spec 定义新点 SaaS 造价系统中的系统用户池专题规则，用于统一平台用户、系统角色、业务身份基础池和项目成员分配前置边界。

## ADDED Requirements

### Requirement: MUST 系统用户池必须作为项目成员分配前置基础池

MUST 系统用户池必须先定义平台级用户、系统角色和基础业务身份，再将其分配到具体项目中。

#### Scenario: 为项目添加成员

- **GIVEN** 项目需要新增成员
- **WHEN** 用户进行成员分配
- **THEN** 应从系统用户池中选择候选用户

### Requirement: MUST 系统角色必须在平台层统一维护

MUST 系统角色属于平台级能力边界，不能在每个项目里随意重新定义同名角色语义。

#### Scenario: 查看用户角色

- **GIVEN** 管理员查看某用户信息
- **WHEN** 系统展示角色配置
- **THEN** 角色应来自平台统一角色体系

### Requirement: MUST 已参与项目的用户不得被无痕删除

MUST 已参与项目或承担历史责任的用户不得被直接无痕删除，而应通过移交、替换或失活方式处理。

#### Scenario: 删除已参项目用户

- **GIVEN** 某用户已参与项目
- **WHEN** 管理员尝试删除该用户
- **THEN** 系统应要求先完成移交、替换或失活

### Requirement: MUST 用户池必须支持业务身份作为补充语义

MUST 除系统角色外，用户池还应支持业务身份作为补充信息，用于后续项目成员授权和业务场景说明。

#### Scenario: 查看用户业务身份

- **GIVEN** 管理员查看某用户资料
- **WHEN** 系统展示用户详情
- **THEN** 应可查看其业务身份信息

## Implementation Notes

以下内容是当前仓库实现现状的观察，不属于规范要求本身：

- 当前仓库在项目成员和权限层已经消费 `roleCode` 等平台角色语义，但未见独立完整的系统用户池模块实现。
- 现有设计文档已明确“系统用户、系统角色和业务身份”是项目成员分配的前置基础池。
- 当前实现更接近“项目成员授权底座已经存在”，而平台级用户管理专题仍主要停留在产品设计层。

这些现状说明系统用户池边界已经明确，但正式用户池管理模块仍属于后续实现内容。
