# Spec: 版本链规则

## 概述

本 spec 定义新点 SaaS 造价系统中清单主链的版本规则，用于约束阶段内引用、跨阶段引用、版本分叉、合同基线锁定、过程调整、来源链追溯和结果重算等稳定业务口径。

本 spec 关注“版本规则本身”，不直接规定具体页面布局或 API 细节。

## ADDED Requirements

### Requirement: 清单必须以版本化对象进行管理

系统中的清单必须以版本化对象进行管理，不同阶段、不同引用动作和不同业务结果必须形成可识别的版本实体，而不是直接覆盖同一份原始数据。

#### Scenario: 创建新的清单成果

- **GIVEN** 用户在某项目阶段内创建或导入一份清单成果
- **WHEN** 该成果进入系统
- **THEN** 系统应为其创建独立版本
- **AND** 该版本应能被后续查询、比较和追溯

### Requirement: 同一业务链上的阶段版本不能被视为数据孤岛

招标清单、投标报价、合同清单、施工过程调整和竣工结算必须被视为同一业务链上的不同版本节点，而不是彼此完全独立的数据孤岛。

#### Scenario: 读取跨阶段链路

- **GIVEN** 用户查看某一阶段的清单版本
- **WHEN** 该版本来源于上游阶段
- **THEN** 系统应能说明它与上游版本的业务关联关系

### Requirement: 跨阶段引用必须生成新的阶段版本

后一阶段引用前一阶段成果时，系统必须生成新的阶段版本，而不能直接在上游版本上继续编辑。

#### Scenario: 从上游阶段引用清单

- **GIVEN** 当前阶段存在可用的上游清单成果
- **WHEN** 用户执行“引用上游版本”
- **THEN** 系统应在当前阶段生成新的版本
- **AND** 新版本应记录其来源版本
- **AND** 上游版本内容不应被本次引用动作直接改写

### Requirement: 引用后的版本必须形成独立分叉

被引用数据进入新阶段后，必须形成独立版本分叉，后续编辑默认只影响当前版本及其下游，不自动回写上游。

#### Scenario: 编辑引用后的版本

- **GIVEN** 某版本由上游成果引用生成
- **WHEN** 用户在当前阶段编辑该版本
- **THEN** 修改应只作用于当前版本
- **AND** 原来源版本应保持不变

### Requirement: 任意版本必须保留完整来源信息

每个版本必须至少保留以下追溯信息：

- 来源版本
- 创建时间
- 创建人
- 变更原因

#### Scenario: 查看版本来源

- **GIVEN** 用户打开某个版本的详情
- **WHEN** 用户查看其版本信息
- **THEN** 系统应展示当前阶段、当前版本号、来源版本、创建人和变更原因等追溯字段

### Requirement: 合同清单必须作为锁定基线管理

合同清单一经确认并锁定，必须作为施工过程和竣工结算的基线对象管理，不允许直接改写其原始基线内容。

#### Scenario: 尝试修改合同基线

- **GIVEN** 某合同清单版本已经成为合同基线
- **WHEN** 用户尝试直接修改该基线版本
- **THEN** 系统应拒绝直接改写该基线内容

### Requirement: 过程调整必须通过独立业务单据体现

设计变更、现场签证、材料认价、进度款等过程调整，必须通过独立业务单据或独立调整版本体现，而不能直接覆盖合同基线。

#### Scenario: 施工过程发生造价调整

- **GIVEN** 项目进入施工过程阶段
- **WHEN** 某项调整影响合同执行金额
- **THEN** 系统应通过变更单、签证单、认价单或其他过程单据记录该调整
- **AND** 合同基线应继续保留为原始基准

### Requirement: 施工过程变更版本必须绑定业务依据

施工过程中的新增、删减和数量调整必须与对应的变更单、签证单或其他过程单据建立可追溯关系。

#### Scenario: 新增变更清单项

- **GIVEN** 用户在施工过程阶段新增或调整清单项
- **WHEN** 该调整将影响造价
- **THEN** 系统应要求其绑定对应业务单据或差异依据

### Requirement: 竣工结算版本必须汇总基线与过程影响

竣工结算版本必须汇总合同基线及全部过程调整结果，形成结算视图版本，并保留与合同基线和过程单据的追溯关系。

#### Scenario: 生成结算版本

- **GIVEN** 项目进入竣工结算阶段
- **WHEN** 系统生成结算成果
- **THEN** 系统应汇总合同基线和已生效的过程调整
- **AND** 结算版本应可追溯至合同基线及相关过程单据

### Requirement: 锁定状态必须限制版本链上的直接写操作

当版本进入锁定态、已提交态或其他只读状态时，系统必须限制对该版本及其从属数据的直接写操作。

#### Scenario: 锁定版本下编辑清单

- **GIVEN** 某清单版本已经锁定或进入不可编辑状态
- **WHEN** 用户尝试修改该版本或其工作内容
- **THEN** 系统应拒绝直接写入

### Requirement: 影响造价的版本变动必须触发关联结果更新

当版本链上的有效变动影响造价结果时，系统必须触发关联汇总、报表、分析或指标结果的更新。

#### Scenario: 过程变动生效后更新结果

- **GIVEN** 某项变更、签证或认价已经生效
- **WHEN** 该事项会影响造价结果
- **THEN** 系统应更新关联汇总、报表或指标结果

### Requirement: 跨项目引用必须受额外控制

历史项目数据可以被引用，但跨项目引用必须受权限与审批控制，不能默认自由复制。

#### Scenario: 引用历史项目成果

- **GIVEN** 用户希望引用历史项目中的成果数据
- **WHEN** 该数据来自其他项目
- **THEN** 系统应在权限和审批规则满足后才允许引用

## Implementation Notes

以下内容是当前仓库实现现状的观察，不属于规范要求本身：

- `apps/api` 当前已有 `BillVersionService`、`BillVersionRepository` 和 `getSourceChain` 等版本链基础能力。
- 当前实现已经支持创建版本、`copyFromVersion` 克隆来源版本以及记录 `sourceVersionId`。
- 当前代码中的版本状态仍相对简化，主要覆盖 `editable`、`submitted`、`locked`。
- 当前实现更接近“版本链骨架”，尚未完整表达文档中更丰富的 `contract_baseline`、`change`、`settlement` 等业务类型口径。

这些现状说明版本链主线已经起步，但本 spec 的业务边界仍应以前述规则为准。
