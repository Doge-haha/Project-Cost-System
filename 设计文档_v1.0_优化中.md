## 一、系统概述

### 1.1 产品定位与设计目标

新点 SaaS 造价系统面向建设工程造价业务，目标不是只解决单一阶段的计价问题，而是将项目从方案概算、招投标、施工跟踪、竣工结算到项目复盘的核心造价数据统一到同一平台中，形成可追溯、可复用、可协作的数据链路。

本系统围绕两个核心目标展开：

1. **打通造价数据全链路流转**
   将投资估算、目标成本、招标清单、投标报价、合同清单、施工过程、竣工结算、项目复盘等数据纳入统一模型管理，减少不同阶段之间依赖文件传递、重复录入和人工核对的问题。
2. **通过 AI 提升计价与审核效率**
   在清单推荐、定额匹配、价格预警、异常审核、材料调差和指标复用等场景中引入 AI 辅助能力，降低人工经验依赖，提高编制和审核质量。

本章用于定义后续章节共同遵守的业务背景、核心概念、阶段模型、角色边界和数据流转规则。除非后文章节明确说明，否则均以本章定义为准。

### 1.2 业务全链路与阶段模型

#### 1.2.1 业务全链路概览

```text
方案概算阶段 → 招投标阶段 → 施工跟踪阶段 → 结算阶段 → 复盘阶段
```

```text
投资估算 → 目标成本 → 招标清单 → 招标控制价 → 投标报价 → 合同清单
→ 施工过程 → 竣工结算 → 项目复盘
```

系统当前统一定义 **9 个业务阶段**，后文涉及“阶段”时，均指以下九项：

| 阶段序号 | 阶段名称 | 所属大阶段 | 说明 |
|---------|---------|-----------|------|
| 1 | 投资估算 | 方案概算 | 根据设计文件和历史指标形成项目初步造价估算 |
| 2 | 目标成本 | 方案概算 | 将估算结果拆分为内部成本控制目标 |
| 3 | 招标清单 | 招投标 | 编制不含价格的招标工程量清单 |
| 4 | 招标控制价 | 招投标 | 在招标清单基础上形成标底或拦标价 |
| 5 | 投标报价 | 招投标 | 投标方基于招标清单形成报价结果 |
| 6 | 合同清单 | 招投标 | 中标后确认并锁定的合同基线清单 |
| 7 | 施工过程 | 施工跟踪 | 围绕合同清单开展变更、签证、计量、进度款等业务 |
| 8 | 竣工结算 | 结算 | 汇总合同执行期全部造价变动，形成结算结果 |
| 9 | 项目复盘 | 复盘 | 提取偏差、指标和经验，沉淀为历史资产 |

为避免后文概念混用，现明确以下边界：

- **阶段**：项目全生命周期中的一级业务环节，用于控制流程顺序、负责人、权限和状态。
- **子工作**：某阶段内部可并行开展的具体业务活动，不单独视为一级阶段。
- **模块**：系统中的功能实现单元，用于支持一个或多个阶段中的业务处理。

因此，施工过程是一个一级阶段；设计变更、现场签证、进度款申报、材料认价、工程计量、变更结算、其他事项属于该阶段下的子工作。

#### 1.2.2 各阶段定义

##### 阶段一：投资估算

| 属性 | 内容 |
|------|------|
| 主责角色 | 造价员 / 估算师 |
| 输入数据 | 设计文件、概算指标、历史项目造价指标 |
| 核心操作 | 根据设计特征套用估算指标，计算主要分项造价 |
| 产出数据 | 投资估算书、项目总造价指标 |
| 下游流转 | 目标成本 |
| AI 辅助 | 推荐相似历史项目指标、提示异常估算区间 |

##### 阶段二：目标成本

| 属性 | 内容 |
|------|------|
| 主责角色 | 项目负责人 / 成本经理 |
| 输入数据 | 投资估算结果、利润率目标、企业内部成本数据库 |
| 核心操作 | 拆分目标成本，形成专业、单体、分部分项级目标值 |
| 产出数据 | 目标成本表、目标单价与目标合价 |
| 下游流转 | 招标清单 |
| AI 辅助 | 对标同类项目，识别目标成本偏高或偏低项 |

##### 阶段三：招标清单

| 属性 | 内容 |
|------|------|
| 主责角色 | 招标方造价员 |
| 输入数据 | 设计文件、目标成本表、工程特征描述 |
| 核心操作 | 按国标清单格式编制工程量清单 |
| 产出数据 | 招标清单文件 |
| 下游流转 | 招标控制价 |
| AI 辅助 | 推荐清单项、检查特征描述完整性 |
| 关键约束 | 招标清单不含价格，仅作为报价依据 |

##### 阶段四：招标控制价

| 属性 | 内容 |
|------|------|
| 主责角色 | 招标方造价员 / 造价咨询单位 |
| 输入数据 | 招标清单、定额库、价目表 |
| 核心操作 | 套定额、计算综合单价，形成标底或拦标价 |
| 产出数据 | 招标控制价成果文件 |
| 下游流转 | 开标后公布或作为限价依据 |
| AI 辅助 | 智能套定额、单价合理性分析、与目标成本偏差分析 |
| 关键约束 | 开标前严格保密，需独立权限控制 |

##### 阶段五：投标报价

| 属性 | 内容 |
|------|------|
| 主责角色 | 投标方造价员 |
| 输入数据 | 招标清单、定额库、价目表、企业内部价格库 |
| 核心操作 | 套定额、测算综合单价、形成投标报价 |
| 产出数据 | 投标报价清单 |
| 下游流转 | 中标后形成合同清单基础版本 |
| AI 辅助 | 推荐定额组合、分析单价区间合理性 |

##### 阶段六：合同清单

| 属性 | 内容 |
|------|------|
| 主责角色 | 项目负责人 |
| 输入数据 | 中标清单、合同约定 |
| 核心操作 | 确认合同清单版本并锁定基线 |
| 产出数据 | 合同清单基线版本 |
| 下游流转 | 施工过程 |
| 关键约束 | 合同清单锁定后不可直接修改，所有调整必须通过后续业务单据体现 |

##### 阶段七：施工过程

| 属性 | 内容 |
|------|------|
| 主责角色 | 施工方造价员 / 项目负责人 |
| 输入数据 | 合同清单、变更指令、签证单、认价单、计量数据 |
| 核心操作 | 在合同执行期间处理造价变动和阶段结算事项 |
| 产出数据 | 变更清单、签证单、进度款申报表、认价单、计量结果、变更结算单 |
| 下游流转 | 竣工结算 |
| AI 辅助 | 自动测算变更造价影响、预警进度偏差和超目标成本项 |
| 关键约束 | 所有涉及造价调整的事项均须经过审批后生效 |

施工过程阶段包含以下子工作：

| 子工作 | 说明 | 主要产出 |
|--------|------|---------|
| 设计变更 | 设计图与合同图纸不符时发起并审批 | 变更清单 |
| 现场签证 | 实际施工与合同约定不符时形成签证 | 签证单 |
| 进度款申报 | 按月或节点汇总已完工程量并申报 | 进度款申报表 |
| 材料认价 | 对无合同单价或需重新确认价格的材料进行认价 | 材料认价单 |
| 工程计量 | 确认已完工程量，作为付款和分析依据 | 已完工程量清单 |
| 变更结算 | 将审批通过的变更金额计入造价结果 | 变更结算单 |
| 其他事项 | 索赔、奖励、罚款等补充性造价事项 | 其他费用单 |

##### 阶段八：竣工结算

| 属性 | 内容 |
|------|------|
| 主责角色 | 造价员 / 审核员 |
| 输入数据 | 合同清单、全部变更清单、签证清单、认价单、竣工图 |
| 核心操作 | 汇总合同执行期全部造价变动，形成结算结果 |
| 产出数据 | 竣工结算书、审核报告 |
| 下游流转 | 项目复盘 |
| AI 辅助 | 对比合同造价与结算造价差异，标识异常项 |

##### 阶段九：项目复盘

| 属性 | 内容 |
|------|------|
| 主责角色 | 项目复盘专员 / 项目负责人 |
| 输入数据 | 全阶段造价数据、实际结算数据、过程偏差数据 |
| 核心操作 | 偏差分析、指标提取、经验归档、知识沉淀 |
| 产出数据 | 项目复盘报告、指标库更新结果、历史造价数据库资产 |
| 关键产出 | 单方造价指标、人材机占比、常用清单与定额组合 |

### 1.3 阶段配置机制

#### 1.3.1 配置原则

并非所有项目都覆盖全部九个阶段。系统支持在项目创建时按模板启用阶段，并对阶段顺序、负责人、权限和完成动作进行独立配置。

项目创建流程如下：

```text
项目创建 → 选择阶段模板 → 调整阶段配置 → 确认负责人和权限 → 进入项目工作台
```

项目级阶段配置遵循以下规则：

1. 阶段启用以项目为单位独立配置。
2. 项目可跳过不适用阶段，但已启用阶段必须定义上下游流转关系。
3. 阶段顺序允许调整，但不得形成环形流转。
4. 阶段负责人按项目维度配置，可随阶段不同而变化。
5. 阶段是否启用 AI 辅助可单独控制。

#### 1.3.2 内置阶段模板

| 模板名称 | 典型客户 | 启用阶段 | 侧重点 |
|---------|---------|---------|--------|
| 全过程工程咨询 | 咨询公司 | 投资估算 → 目标成本 → 招标清单 → 招标控制价 → 投标报价 → 合同清单 → 施工过程 → 竣工结算 → 项目复盘 | 覆盖全生命周期 |
| 过程跟踪审计 | 建设单位、审计机构 | 合同清单 → 施工过程 → 竣工结算 | 重点监控施工阶段造价变化 |
| 造价咨询（标底编制） | 咨询公司、招标方 | 招标清单 → 招标控制价 → 投标报价 | 重点在清单准确性和控制价编制 |
| 结算审核 | 审计机构、造价咨询公司 | 竣工结算 | 重点在竣工后造价审核 |
| 自定义 | 任意客户 | 从 9 个阶段中自由组合 | 按项目实际需要灵活配置 |

说明：

- 上表为系统内置模板。
- 若后文示例出现“招标代理”“施工预算”等名称，均视为基于“自定义”模板衍生出的业务场景，不单独作为内置模板定义。

#### 1.3.3 阶段配置项

| 配置项 | 说明 |
|--------|------|
| 启用阶段 | 从 9 个阶段中选择需要的阶段 |
| 阶段顺序 | 配置启用阶段的前后流转关系 |
| 阶段负责人 | 为每个阶段指定负责人 |
| 阶段数据权限 | 定义该阶段可见、可引用、可编辑的数据范围 |
| 阶段完成动作 | 自动触发下一阶段、人工确认后流转或归档 |
| AI Copilot 开关 | 控制阶段是否启用 AI 辅助能力 |

#### 1.3.4 阶段配置示例

**案例一：招标代理机构项目**

```yaml
项目: 江苏XX广场招标
模板: 自定义（招标代理场景）
阶段配置:
  - 阶段: 招标清单
    负责人: @造价员A
    AI启用: true
    完成后: 自动通知下一阶段负责人
  - 阶段: 投标报价
    负责人: @造价员B
    AI启用: true
    完成后: 生成合同清单初始版本
跳过阶段:
  - 投资估算
  - 目标成本
  - 施工过程
  - 竣工结算
  - 项目复盘
```

**案例二：施工单位全链路项目**

```yaml
项目: XX项目施工预算
模板: 自定义（施工预算场景）
阶段配置:
  - 阶段: 投标报价
    负责人: @预算员A
  - 阶段: 合同清单
    负责人: @预算员A
    锁定: true
  - 阶段: 施工过程
    负责人: @预算员B

```

### 1.4 核心业务对象与数据流转规则

#### 1.4.1 核心业务对象

后文涉及的数据对象以本节定义为准。

| 对象 | 定义 | 说明 |
|------|------|------|
| 项目 | 系统中的最小业务管理单元 | 对应一个独立建设项目或专项造价任务 |
| 阶段 | 项目生命周期中的一级业务环节 | 由项目模板或自定义配置启用 |
| 专业方向 | 项目中的专业分类维度 | 如土建、安装、装饰、园林、市政等 |
| 工程单体 | 专业方向下的独立核算单元 | 如 1#楼、地下车库、室外管网等 |
| 清单 | 造价业务中的核心明细载体 | 可存在于不同阶段，具备版本属性 |
| 合同清单 | 中标后锁定的清单基线版本 | 后续调整不能直接改写该对象 |
| 变更清单 | 由设计变更形成的清单调整对象 | 与合同清单建立关联关系 |
| 签证单 | 由现场实际情况形成的补充造价对象 | 需审批后生效 |
| 材料认价单 | 对材料单价进行确认的业务单据 | 用于影响计价结果 |
| 结算书 | 汇总合同及过程变动后的最终成果对象 | 作为结算和审计依据 |
| 指标库记录 | 从项目复盘提取的结构化指标资产 | 可供新项目引用 |

#### 1.4.2 组织维度与汇总层级

项目内部按“专业方向 × 工程单体”进行分工和汇总。

```text
项目总造价 = Σ(各专业方向造价)
专业方向造价 = Σ(该专业下各工程单体造价)
工程单体造价 = 分部分项费 + 措施费 + 其他费 + 规费 + 税金
```

示意结构如下：

```text
项目
├── 专业方向
│   ├── 工程单体A
│   ├── 工程单体B
│   └── 工程单体C
└── 专业方向
    ├── 工程单体D
    └── 工程单体E
```

#### 1.4.3 数据引用、分叉与锁定规则

系统中的跨阶段数据流转遵循以下规则：

| 规则 | 说明 |
|------|------|
| 阶段内引用 | 同一阶段内可直接引用已有清单和关联数据 |
| 跨阶段引用 | 后一阶段可引用前一阶段结果，但引用后形成新的阶段版本 |
| 版本分叉 | 被引用数据在新阶段内形成独立版本，不反向覆盖来源阶段数据 |
| 合同锁定 | 合同清单一经锁定，不允许直接修改原始基线内容 |
| 过程调整 | 变更、签证、认价等只通过独立业务单据体现，不改写合同基线 |
| 结果重算 | 影响造价的变动生效后，关联汇总、报表、指标自动重算 |
| 跨项目引用 | 历史项目数据可被引用，但需受权限和审批控制 |

进一步约束如下：

1. 招标清单、投标报价、合同清单属于同一业务链上的不同阶段版本，不视为完全独立的数据孤岛。
2. 合同清单是施工过程和竣工结算的基线对象，后续所有造价变化必须可追溯到具体业务单据。
3. 任意版本必须保留来源关系、创建时间、创建人和变更原因。
4. 引用产生的新版本默认仅影响当前阶段及其下游阶段，不自动回写上游阶段。

#### 1.4.4 数据壁垒问题与系统解决思路

| 典型问题 | 系统设计思路 |
|---------|-------------|
| 招标清单和投标报价分离，依赖人工比对 | 使用统一清单模型，在不同阶段按不同视图展示 |
| 施工过程中的变更无法及时联动到结算 | 建立基于合同清单的变更关联关系，自动重算汇总与结算 |
| 不同参与方数据彼此隔离，协同成本高 | 在同一平台上通过权限和阶段视图隔离数据 |
| 审核严重依赖人工经验，容易漏项 | 通过 AI 辅助审核和异常预警提高识别效率 |

### 1.5 角色、组织与权限边界

#### 1.5.1 系统角色

| 角色 | 职责范围 |
|------|---------|
| 系统管理员 | 用户管理、权限配置、基础数据维护 |
| 项目负责人 | 项目创建、阶段配置、成员管理、跨阶段总览 |
| 造价员 | 清单编制、定额套用、计价处理、过程单据维护 |
| 审核员 | 审核业务成果、查看异常、执行审核结论 |
| 项目复盘专员 | 提取指标、沉淀历史经验、维护指标资产 |

#### 1.5.2 业务身份说明

同一个“系统角色”在不同业务场景下可能具有不同业务身份。为避免后文权限设计过粗，定义如下：

| 业务身份 | 典型阶段 |
|---------|---------|
| 招标方造价员 | 招标清单、招标控制价 |
| 投标方造价员 | 投标报价 |
| 施工方造价员 / 预算员 | 合同清单、施工过程、竣工结算 |
| 审计方审核员 | 招标控制价审核、结算审核 |

#### 1.5.4 分工规则

| 规则 | 说明 |
|------|------|
| 阶段独立分配 | 同一单体在不同阶段可由不同人员负责 |
| 单体独立负责 | 同一专业下不同单体可由不同人负责 |
| 跨专业兼任 | 同一人可负责多个专业或单体 |
| 汇总可见 | 项目负责人可查看全项目汇总视图 |

## 三、清单管理模块

### 3.1 清单编制页（核心页面）

适用阶段：招标清单、投标报价、合同清单、施工过程中的变更清单、竣工结算。

页面目标：完成清单的新增、引用、版本分叉、特征补充、规则校验和阶段提交，是全系统最核心的数据录入与维护页面。

#### 3.1.1 页面结构

| 区域 | 功能 |
|------|------|
| 顶部工具栏 | 提供新增、导入导出、搜索、AI 推荐、校验、保存等操作 |
| 左侧章节树 | 按章节、节、子目浏览当前清单结构 |
| 中央表格区 | 展示清单主数据并支持编辑 |
| 右侧详情侧栏 | 展示当前清单项的特征、计算规则、来源和操作记录 |
| 底部状态栏 | 展示保存状态、校验结果、当前版本信息 |

#### 3.1.2 阶段差异说明

同一套清单模型在不同阶段使用同一页面，但字段可编辑范围和数据来源不同。

| 阶段 | 页面用途 | 可编辑重点 | 关键限制 |
|------|---------|-----------|---------|
| 招标清单 | 编制招标工程量清单 | 编号、名称、单位、工程量、特征描述 | 不允许填写正式价格 |
| 投标报价 | 基于招标清单形成报价版本 | 工程量引用、定额关联、综合单价 | 需保留来源招标清单版本 |
| 合同清单 | 锁定中标后的合同基线 | 基线确认、备注补充 | 锁定后禁止直接改写 |
| 施工过程变更清单 | 记录设计变更或签证引起的清单调整 | 新增/删减项、调整数量、变更原因 | 必须绑定变更单或签证单 |
| 竣工结算 | 汇总合同执行期结果 | 结果核对、差异说明 | 需保留与合同基线及过程单据的追溯关系 |

#### 3.1.3 版本与来源展示

清单页必须显式展示当前版本来源，避免用户误把不同阶段版本当成同一份可直接覆盖的数据。

| 展示项 | 说明 |
|------|------|
| 当前阶段 | 当前清单所属阶段 |
| 当前版本号 | 如 v1.0 / v2.0 / v3.0 |
| 来源版本 | 引用自哪个阶段、哪个版本 |

#### 3.1.7 校验规则

1. 清单编号不能为空，且同层级内不得重复。

- 提交、驳回、通过均需记录操作日志，并在版本记录中可追溯。

## 五、计价引擎模块

### 5.1 综合单价计算规则

本章定义系统内综合单价、合价、费用分摊和重算触发机制，是清单管理、定额管理、汇总报表和审核流程的统一计算基础。

#### 5.1.1 计价引擎目标

计价引擎负责完成以下工作：

1. 根据清单、定额、价目表和取费设置计算综合单价。
2. 将措施费、规费、税金按规则分摊到项目、专业、单体或清单。
3. 响应用户编辑、导入、引用和审核动作，自动触发重算。
4. 保留系统计算值、人工调整值和审计日志，支撑复核与追溯。

**计算流程（自动执行）：**

```text
综合单价 = 分部分项单价 + 措施项目分摊单价 + 规费分摊单价 + 税金分摊单价
其中：
  分部分项单价 = Σ(定额合价 × 含量系数) / 清单工程数量
  定额合价 = 工程量 × 定额单价
  措施项目分摊单价 = 措施项目费 / 对应清单工程数量或分摊基数
  规费分摊单价 = 规费 / 对应清单工程数量或分摊基数
  税金分摊单价 = 税金 / 对应清单工程数量或分摊基数
```

说明：

- V1 采用可配置的分摊规则，将措施费、规费、税金按项目、专业或单体维度分摊。
- 若某阶段不允许人工调价，则综合单价完全由引擎计算得出。
- 若某阶段允许人工调价，系统需保留“系统计算值”和“人工调整值”两套字段用于审计。

#### 5.1.2 计价结果字段

| 字段 | 说明 |
|------|------|
| 系统计算综合单价 | 引擎按当前规则计算的结果 |
| 人工调整综合单价 | 用户在允许阶段下手工调整后的结果 |
| 最终生效综合单价 | 当前阶段对外使用的综合单价 |
| 系统计算金额 | 系统计算综合单价 × 工程数量 |
| 最终生效金额 | 最终生效综合单价 × 工程数量 |
| 最近计算时间 | 最近一次引擎计算完成时间 |
| 计算来源 | 自动计算 / 手动触发 / 批量重算 |

#### 5.1.3 分摊策略

V1 的分摊策略应支持至少以下几种模式：

| 分摊项 | 支持策略 | 说明 |
|------|---------|------|
| 措施费 | 按项目 / 专业 / 单体 / 清单分摊 | 用于匹配不同业务场景 |
| 规费 | 按人工费基数或定额费用基数分摊 | 需与取费模板一致 |
| 税金 | 按税率统一计算后分摊 | 与税率类别和地区规则关联 |

若项目未配置特殊规则，则默认采用项目级统一分摊策略。

#### 5.1.4 重算触发机制

**自动计算触发时机：**
1. 用户修改清单工程数量时
2. 用户修改/新增/删除关联定额时
3. 用户调整含量系数时
4. 用户调整取费费率时
5. 用户点击「重新计算」按钮时

补充触发场景：

6. 用户切换价目版本或地区时
7. 用户提交上游阶段引用并生成新版本时
8. 审核驳回后重新编辑影响价格字段时
9. 批量导入清单或定额完成后

#### 5.1.5 计算范围控制

| 触发方式 | 默认重算范围 |
|---------|-------------|
| 单行字段编辑 | 当前清单项 |
| 定额明细变更 | 当前清单项及所属汇总 |
| 取费参数调整 | 当前专业、单体或项目全部受影响项 |
| 价目版本切换 | 当前阶段全部受影响项 |
| 批量重算 | 当前项目或指定专业/单体范围 |

#### 5.1.6 人工调价与审计规则

- 仅在允许人工调价的阶段开放人工调整入口。
- 人工调价后不得覆盖系统计算值，必须保留原始结果。
- 人工调价必须填写调整原因，可选绑定审批意见。
- 汇总报表和审核页面应同时支持查看“系统值”“人工调整值”“最终生效值”。
- 若人工调整值偏离系统值超过预设阈值，应触发 AI 预警或审核提醒。

| 备注 | 文本 | — | — |

#### 8.2.1 管理规则

- 清单库以国标清单编码为核心主键，允许按地区扩展项目特征。
- 标准价格仅作为参考值，不直接作为项目正式计价结果。
- 已被项目引用的清单模板修改后，不自动回写历史项目。
- AI 清单推荐应优先返回当前项目类型、专业和地区匹配的清单项。

### 8.3 价目表管理

页面目标：维护不同地区、不同时间版本下的取价数据，为定额单价和综合单价计算提供价格基础。

| 按键/区域 | 功能 |

| 📥 批量更新价目 | 从Excel批量更新价目表 |

|------|------|------|------|
| 版本号 | 文本 | ✅ | 如：2024Q4 |
| 版本日期 | 日期 | ✅ | 如：2024-12-01 |
| 所属地区 | 下拉 | ✅ | 价格所属地区 |
| 定额编号 | 文本 | ✅ | 关联定额库 |

#### 8.3.1 管理规则

页面目标：建立平台级、项目级、阶段级、专业级和单体级的多层权限体系。

**功能权限矩阵：**

| 功能 | 系统管理员 | 项目负责人 | 造价员 | 审核员 | 项目复盘专员 |
|------|-----------|-----------|--------|--------|---------------|
| 系统设置 | ✅ | ❌ | ❌ | ❌ | ❌ |

- 业务身份决定项目内阶段级、数据级权限。

- 若项目阶段已锁定，则即使用户有编辑权限，也不得直接修改锁定基线。

## 十、技术接口设计（API）

### 10.1 REST API 结构

接口设计目标：让前端、第三方系统和后续 AI 服务通过统一资源模型访问项目、阶段、清单、定额、汇总和基础数据。

```text
POST   /api/v1/auth/login          # 登录
POST   /api/v1/auth/logout         # 登出
GET    /api/v1/auth/me             # 当前用户信息

GET    /api/v1/projects            # 项目列表
POST   /api/v1/projects            # 新建项目
GET    /api/v1/projects/:id        # 项目详情
PUT    /api/v1/projects/:id        # 更新项目
DELETE /api/v1/projects/:id        # 删除项目
GET    /api/v1/projects/:id/stages         # 项目阶段配置
PUT    /api/v1/projects/:id/stages         # 更新项目阶段配置

GET    /api/v1/projects/:id/bill-items             # 项目清单
POST   /api/v1/projects/:id/bill-items             # 添加清单行
PUT    /api/v1/projects/:id/bill-items/:itemId     # 更新清单
DELETE /api/v1/projects/:id/bill-items/:itemId     # 删除清单

GET    /api/v1/projects/:id/quota-lines            # 项目定额
POST   /api/v1/projects/:id/quota-lines            # 添加定额行
PUT    /api/v1/projects/:id/quota-lines/:lineId    # 更新定额行
DELETE /api/v1/projects/:id/quota-lines/:lineId    # 删除定额行

POST   /api/v1/projects/:id/review-submissions     # 提交当前阶段审核
POST   /api/v1/projects/:id/change-orders          # 发起变更
POST   /api/v1/projects/:id/site-visas             # 发起签证
POST   /api/v1/projects/:id/progress-payments      # 发起进度款申报

POST   /api/v1/engine/calculate      # 计算综合单价（单条）
POST   /api/v1/engine/calc-all       # 重算全项目

GET    /api/v1/reports/summary       # 造价汇总
GET    /api/v1/reports/bill-items/:id  # 清单报表
POST   /api/v1/reports/export        # 导出报表

GET    /api/v1/libraries/quotas       # 定额库
POST   /api/v1/libraries/quotas       # 新增定额
GET    /api/v1/libraries/bill-templates  # 清单库
POST   /api/v1/libraries/bill-templates  # 新增清单模板

GET    /api/v1/libraries/prices      # 价目表
POST   /api/v1/libraries/prices      # 更新价目
```

#### 10.1.1 接口设计规则

- 资源命名使用复数名词，保持 REST 风格一致。
- 阶段、版本、地区等筛选条件优先通过 query 参数传递。
- 所有写接口均需记录操作者、时间和业务上下文，支持审计追溯。
- 合同清单锁定后的修改类接口应返回受限状态码，并提示改走过程单据接口。
- 导出类接口建议采用异步任务模式，避免大文件导出阻塞请求。

#### 10.1.2 推荐补充接口

| 接口 | 用途 |
|------|------|
| `GET /api/v1/projects/:id/summary/variance` | 获取项目偏差分析 |
| `GET /api/v1/projects/:id/version-chain` | 获取项目版本链 |
| `GET /api/v1/projects/:id/audit-logs` | 获取项目审计日志 |
| `POST /api/v1/ai/bill-recommendations` | AI 清单推荐 |
| `POST /api/v1/ai/quota-recommendations` | AI 定额推荐 |

### 10.2 数据库选型建议

| 用途 | 建议方案 | 说明 |
|------|---------|------|
| 主数据库 | PostgreSQL | 支持JSON、全文检索、事务完整 |
| 缓存层 | Redis | 项目会话缓存、计算结果缓存 |
| 文件存储 | 本地MinIO / 阿里云OSS | 报表文件、导入导出文件 |
| 定额库搜索 | PostgreSQL pgvector | 支持向量相似度搜索（未来AI辅助套定额） |

#### 10.2.1 数据存储建议

- 业务主数据建议按“项目主表 + 阶段配置表 + 清单版本表 + 定额明细表 + 过程单据表”拆分。
- 审计日志建议独立建表，避免与业务主表耦合。
- 报表文件和导入文件建议只存元数据到数据库，实体文件走对象存储。
- AI 推荐结果建议作为辅助结果存储，不直接覆盖正式业务数据。

## 十一、字段命名规范

本章用于统一数据库、接口和前端字段命名，避免后续开发出现同义字段、多套命名和跨层口径不一致。

```text
数据库字段命名：snake_case（小写下划线）
API 请求/响应：camelCase（JS风格）
前端组件：PascalCase

表名前缀：
  prj_       — 项目相关
  stg_       — 阶段配置相关
  bil_       — 清单相关
  qte_       — 定额相关
  chg_       — 变更/签证/过程单据相关
  rpt_       — 报表相关
  lib_       — 数据字典/库
  sys_       — 系统配置
```

#### 11.1 推荐命名原则

- 同一业务对象在数据库、接口和前端中应保持语义一致，仅风格转换不改业务含义。
- 阶段名称、版本号、地区编码等通用字段应统一复用，不重复造词。
- “bill”统一表示清单项资源，“quota”统一表示定额资源，“price”统一表示价目资源。
- 人工调整相关字段统一采用 `manual_` 前缀，系统计算字段统一采用 `system_` 前缀。

## 十二、状态码与错误处理

本章用于统一接口返回、页面反馈、批量导入和计算任务过程中的错误处理方式。

### 操作反馈（前端 toast/alert）

| 状态码 | 提示文案 | 级别 |
|--------|---------|------|
| 0 / 200 | 操作成功 | success（绿色） |
| 400 | 参数错误：{具体字段} | warning（黄色） |
| 401 | 请先登录 | error（红色） |
| 403 | 无权限操作，请联系管理员 | error（红色） |
| 404 | 数据不存在或已删除 | warning（黄色） |
| 409 | 数据冲突：{冲突原因} | warning（黄色） |
| 500 | 服务器异常，请稍后重试 | error（红色） |

#### 12.1 推荐补充状态码

| 状态码 | 适用场景 |
|------|---------|
| 422 | 字段校验通过但业务规则不满足 |
| 423 | 数据处于锁定状态，不允许直接修改 |
| 202 | 异步导出、批量重算等任务已受理 |

### 表格数据状态

| 状态 | 前端表现 |
|------|---------|
| 正常 | 白色行 |
| 已修改未保存 | 浅黄(#FFFEF0)行 + 行首显示 ● 图标 |
| 保存失败 | 浅红(#FFF0F0)行 + 行首显示 ❌ 图标 |
| 计算中 | 数字字段显示 spinner 动画 |
| 计算失败 | 显示 "#ERR" 红色文字 |

#### 12.2 任务型错误处理

| 场景 | 处理方式 |
|------|---------|
| 批量导入失败 | 返回失败明细并允许下载错误报告 |
| 批量重算失败 | 标识失败范围并允许重试 |
| 报表导出失败 | 保留任务记录并展示失败原因 |
| AI 推荐失败 | 不阻断人工操作，提示稍后重试 |
